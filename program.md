# AutoStock — 日本株スコアリングモデル自律実験

## セットアップ

実験を始める前に、以下の手順を完了させよ。

1. **ランタグの決定**: 今日の日付ベースでタグを提案せよ（例: `apr2`）。ブランチ `autostock/<tag>` が既存でないことを確認する。
2. **ブランチ作成**: `git checkout -b autostock/<tag>`
3. **ファイルの通読**: 以下の3ファイルを必ず読み、全体像を頭に入れよ。
   - `program.md` — 実験ルール（このファイル）。変更不可。
   - `prepare.py` — データ取得・特徴量生成・評価関数。**変更不可（FIXED）**。
   - `train.py` — スコアリングモデル・訓練ループ。**これだけを編集する。**
4. **データ確認**: `data/ohlcv/` にキャッシュ済みCSVが存在することを確認。なければ人間に `python prepare.py --all` の実行を依頼せよ。
5. **results.tsv の初期化**: ヘッダー行のみで `results.tsv` を作成する（後述のフォーマット）。
6. **確認して開始**: セットアップ完了を確認したら、実験ループに入る。

## 目標

日本株（東証プライム・スタンダード）の中から、40営業日後に値上がりする確率が高い銘柄を見つけるスコアリングモデルを構築する。

**ゴールはシンプル: `top20_return` を最大化せよ。**

## 評価指標

`=== RESULT ===` ブロックの値で判断する。

| 指標 | 意味 | 判断基準 |

## 禁止事項

1. **prepare.py の変更** — 評価関数・特徴量生成は固定
2. **未来データのリーク** — テスト期間（2024-01-01以降）の情報を訓練に使わない（詳細は後述）
3. **テスト期間での過学習** — テスト結果を見てHPを手動チューニングする行為
4. **外部パッケージの追加** — lightgbm, pandas, numpy, scikit-learn のみ使用可
5. **実行時間5分超過** — `python train.py` が5分以内に完了すること

## 未来データリーク：具体的に何がダメか

株価予測で最も致命的なバグは「未来の情報が訓練に漏れる」こと。以下は全て禁止。

| リークの種類 | 具体例 | なぜダメか |
|-------------|--------|-----------|
| **直接リーク** | テスト期間（2024年以降）のデータを訓練セットに含める | 答えを見て勉強しているのと同じ |
| **正規化リーク** | 訓練+テスト全体の平均・標準偏差で特徴量を正規化する | テスト期間の統計情報が訓練に漏れる |
| **ターゲットリーク** | ターゲット（将来リターン）から派生した特徴量を作る | 将来の株価を知っているのと同じ |
| **ルックアヘッドバイアス** | 「今日のデータ」として未来に修正された値を使う（決算修正値など） | 実運用時には手に入らない情報 |
| **サバイバーシップバイアス** | 上場廃止銘柄を除外してバックテストする | 生き残った銘柄だけ見ると成績が良く見える |
| **クロスセクションリーク** | 日付Tで「その日の全銘柄の情報」を使ってランキング→それ自体は良いが、ランキング計算にテスト期間のデータが混入 | 微妙だが注意が必要 |

**安全な特徴量の原則**: その特徴量は「その日の引け後に本当に手に入る情報だけで計算できるか？」をチェックせよ。

## 過学習の兆候と対処

以下のサインが出たら過学習を疑え。

| サイン | 意味 | 対処 |
|--------|------|------|
| 訓練リターンが非常に高いのにテストリターンが低い | モデルが訓練データの固有パターンを暗記している | 正則化を強める（`lambda_l1/l2`増加、`min_child_samples`増加、`num_leaves`削減） |
| num_boost_roundが上限まで使い切っている | early stoppingが効いていない可能性 | 検証分割の設計を見直す |
| 特徴量重要度が1つに極端に偏っている | その特徴量に過剰依存している | `feature_fraction`を下げる、その特徴量をノイズ化してテスト |
| 実験を重ねるたびに少しずつ「テスト結果を見て判断」していて微改善が続く | テストセットへの間接的な過学習（information leakage through experimenter） | 定期的にベースラインに戻して改善幅が本物か確認する |

**参考表示**: train.py のStep 6「Reference: Evaluation on train period」とStep 5のテスト評価を比較せよ。差が大きすぎるなら過学習。

## 実験ループ

### 初回実行

最初の実行は必ず **ベースラインの確立** に使え。train.py を一切変更せずに実行し、結果を記録する。

### ループ本体

**LOOP FOREVER:**

1. **git状態の確認**: 現在のブランチ・コミットを確認する
2. **train.py を編集**: 1つの仮説を検証する変更を加える（**1回の実験で変えるのは1つだけ**）
3. **git commit**: 変更をコミットする（実行前に必ず）
4. **実験実行**: `python train.py > run.log 2>&1`（出力をリダイレクト。teeは使うな。コンテキストが溢れる）
5. **結果の抽出**: `grep "top20_return\|excess_return\|sharpe_ratio" run.log`
6. **クラッシュ判定**: grepの出力が空なら、クラッシュしている。`tail -n 50 run.log` でスタックトレースを確認し修正を試みる。
7. **results.tsv に記録**: 結果を追記する（後述のフォーマット）
8. **判断**:
   - `top20_return` が改善 → **keep**: ブランチをそのまま進める
   - `top20_return` が悪化 or 同等 → **discard**: `git reset --hard HEAD~1` で前のコミットに戻す
   - crash → 修正を試みる。2回失敗したら **crash** として記録し、`git reset --hard HEAD~1`

### 「1回の実験で変えるのは1つだけ」の原則

これは科学実験の基本。破ると何が効いたか分からなくなる。

- **良い例**: `learning_rate` を 0.05 → 0.1 に変更する（他は全て同じ）
- **良い例**: 特徴量 `ret_5d_sector_rel` を除外する（他は全て同じ）
- **悪い例**: `learning_rate` を変えつつ `num_leaves` も変えつつ特徴量も追加する
- **例外**: 論理的に不可分な変更はセットでOK（例: 新しい特徴量を追加 + その特徴量を `SELECTED_FEATURES` に含める）

### NEVER STOP

実験ループが開始されたら、**人間に「続けますか？」と聞くな**。人間は寝ているかもしれない。お前は自律研究者だ。アイデアが尽きたら、以下を試せ:

- results.tsv を読み返して、過去の実験パターンから新しい仮説を立てる
- 過去のnear-miss（僅差で改善しなかった実験）を組み合わせる
- 全く異なるアプローチに切り替える（Phase を飛ばしてOK）
- 後述の「行き詰まったときのアイデアバンク」を参照する

ループは人間が手動で止めるまで続く。

### モデル使い分けルール

- **通常のループ（HP変更、特徴量除外、小さな改善）** → **Sonnet** を使え（高速・低コスト）
- **アイデアが尽きたとき、根本的な方針転換が必要なとき** → **Opus** に切り替えて深く考えろ
  - 具体的には: 10回以上連続で discard が続いた場合、Phase を跨ぐ判断が必要な場合、全く新しいアプローチを設計する場合
  - Opus で方針を決めたら、実行はまた Sonnet に戻す

## results.tsv のフォーマット

タブ区切り（TSV）。カンマは説明文に使うことがあるのでカンマ区切りにしない。results.tsv は **git commit しない**（untracked のまま）。

ヘッダーと5列:

```
commit	top20_return	sharpe	status	description
```

| 列 | 内容 |
|----|------|
| commit | git commit hash（短縮7文字） |
| top20_return | 上位20銘柄の平均リターン（例: 0.0523）。クラッシュ時は 0.0000 |
| sharpe | Sharpe Ratio（例: 1.23）。クラッシュ時は 0.00 |
| status | `keep`, `discard`, `crash` のいずれか |
| description | 何を試したかの短い説明 |

例:

```
commit	top20_return	sharpe	status	description
a1b2c3d	0.0523	1.23	keep	baseline
b2c3d4e	0.0612	1.45	keep	increase learning_rate to 0.1
c3d4e5f	0.0498	1.10	discard	switch to huber loss
d4e5f6g	0.0000	0.00	crash	double num_leaves (memory error)
```

## train.py の改変ガイド

train.py は以下のセクションに分かれている。セクションごとの改変方針を示す。

### [Hyperparameters] セクション

最もリスクが低く、最も頻繁に変更するセクション。

```python
LGB_PARAMS = {
    "num_leaves": 31,        # 7〜127 の範囲で試す。大きいほど表現力↑ 過学習リスク↑
    "learning_rate": 0.05,   # 0.01〜0.2 の範囲。小さいほど安定、大きいほど高速
    "feature_fraction": 0.8, # 0.5〜1.0。低いほど正則化効果↑
    "bagging_fraction": 0.8, # 0.5〜1.0。低いほどノイズ耐性↑
    "min_child_samples": 20, # 5〜100。大きいほど保守的
    "lambda_l1": 0.1,        # 0〜10。L1正則化
    "lambda_l2": 0.1,        # 0〜10。L2正則化
}
NUM_BOOST_ROUND = 500        # 100〜2000
EARLY_STOPPING_ROUNDS = 50   # 20〜100
```

**ヒント**: 最初は `num_leaves` と `learning_rate` の組み合わせだけで大きく変わる。

### [Feature Selection] セクション

特徴量の選択・除外を行う。

```python
SELECTED_FEATURES = None  # None=全特徴量。リスト指定で絞り込み
```

**戦略**:
- まず `None`（全特徴量）でベースラインを作る
- feature importance を見て、重要度ゼロの特徴量を除外する
- ノイズになっている特徴量を1つずつ除外して効果を確認する

### [Model Training] セクション

`train_model()` 関数の中身。ここを変えるとモデルの訓練方法自体が変わる。

**変更例**:
- 訓練/検証の分割比率（0.8 → 0.75 等）
- ターゲット変数の変換（ランク化、クリッピング）
- サンプル重み付け（直近データを重視）
- 損失関数の変更（regression → lambdarank 等）
- 複数モデルのアンサンブル

### [Prediction] セクション

`predict_scores()` 関数。通常は変更不要だが、アンサンブル時に変更が必要。

### [Main] セクション

実行フロー。基本的に触らない。評価の呼び出し方を変えたい場合のみ。

## 探索の方向性（フェーズ別）

### Phase 1: ハイパーパラメータ探索（最初にやれ）

ベースラインを確立した後、まずHPだけで改善できる範囲を探る。

**具体的に試すこと**:
1. `learning_rate`: 0.01, 0.03, 0.05, 0.1, 0.2
2. `num_leaves`: 15, 31, 63, 127
3. `feature_fraction`: 0.5, 0.6, 0.7, 0.8, 1.0
4. `min_child_samples`: 5, 10, 20, 50, 100
5. `bagging_fraction`: 0.6, 0.7, 0.8, 0.9
6. `NUM_BOOST_ROUND`: 200, 500, 1000, 2000（early stoppingがあるので上限を上げるだけでもOK）
7. `lambda_l1 / lambda_l2`: 0, 0.01, 0.1, 1.0, 5.0

**HP探索の順序**: learning_rate → num_leaves → 正則化パラメータ → その他

### Phase 2: 特徴量エンジニアリング（最も改善が見込める）

prepare.py が生成する特徴量の「使い方」を最適化する。prepare.py自体は変えないが、train.py内での前処理・選択・変換は自由。

**具体的に試すこと**:
1. **不要特徴量の除外**: feature importance が低い特徴量を除外する
2. **特徴量のランク化**: 数値をクロスセクションのランク（percentile）に変換する
   ```python
   # 日付ごとにランク化する例
   X_train = X_train.groupby(level="Date").rank(pct=True)
   ```
3. **特徴量のWinsorize**: 外れ値を1%ile/99%ileでクリッピングする
4. **交差特徴量**: モメンタム x ボラティリティ、RSI x 出来高比率 など
5. **ラグ特徴量**: 5日前のRSI、20日前のモメンタム等（prepare.pyが出す特徴量のshift）
6. **差分特徴量**: 今日のRSI - 5日前のRSI（モメンタムの加速度）
7. **セクター相対特徴量の活用**: `INCLUDE_SECTOR_REL = True` の効果を検証

### Phase 3: ターゲット変数の工夫

ターゲット（40日後リターン）の扱い方を変える。

**具体的に試すこと**:
1. **ターゲットのランク化**: リターン値の代わりにクロスセクショナルランクを予測する
   ```python
   y_train = y_train.groupby(level="Date").rank(pct=True)
   ```
2. **ターゲットのクリッピング**: 極端なリターン（±50%等）をクリッピング
3. **ターゲットの二値化**: 上位N%を1、それ以外を0として分類問題にする
4. **サンプル重み付け**: 直近年のデータに高い重みを付ける（市場レジーム変化への適応）
   ```python
   weights = np.where(dates.year >= 2022, 2.0, 1.0)
   ```

### Phase 4: モデル構造の変更

LightGBMの使い方を根本的に変える、または複数モデルを組み合わせる。

**具体的に試すこと**:
1. **損失関数の変更**: `objective` を `huber`, `quantile`, `lambdarank` に変更
2. **時系列CV**: 訓練期間を年ごとに分割してCVし、各fold のモデルの予測を平均する
3. **複数forward_daysのアンサンブル**: 20日/40日/60日のモデルを作り、スコアを平均
4. **セクター中立化**: セクターごとにモデルを分ける、またはスコアからセクター平均を引く
5. **スタッキング**: LightGBM + Ridge回帰のブレンド
6. **カスタム目的関数**: 上位銘柄のリターンを最大化するように直接最適化

### Phase 5: 高度な手法（行き詰まったとき）

1. **Adversarial Validation**: 訓練とテストの分布が異なる特徴量を特定し除外
2. **Feature interaction制約**: LightGBMの `interaction_constraints` で特徴量の組み合わせを制限
3. **Monotone constraints**: 「RSIが高いほどスコアが低い」等の単調性制約
4. **Target encoding**: セクターや規模の情報をターゲットの統計量でエンコード（リーク注意）
5. **Purged cross-validation**: 訓練と検証の間にギャップ期間を設けてリークを防ぐ

## 行き詰まったときのアイデアバンク

実験を20回以上回して改善が止まったら、以下を試せ。

### 視点を変える系
- [ ] 全特徴量をランク化してからモデルに入れる（外れ値の影響を除去）
- [ ] ターゲットをリターンではなくランクにする（順序予測に切り替え）
- [ ] `objective` を `regression` から `lambdarank` に変更する
- [ ] 逆転の発想: bottom銘柄を避けるモデルを作り、スコアを反転させる

### 過学習を疑う系
- [ ] `num_leaves` を半分にする
- [ ] `min_child_samples` を 100 に上げる
- [ ] `feature_fraction` を 0.5 に下げる
- [ ] 訓練期間を短くする（2019年以降のみ等 — 最近のレジームに絞る）
- [ ] early_stopping の検証期間を変える（最後20% → 最後30%）

### 特徴量を見直す系
- [ ] feature importance がゼロの特徴量を全部除外する
- [ ] 上位5特徴量だけでモデルを作る（シンプルなモデルの方が強い場合がある）
- [ ] 相関が高い特徴量ペアの片方を除外する
- [ ] 特徴量を全てz-score正規化してから入れる（日付ごとにクロスセクションで正規化）

### アンサンブル系
- [ ] 異なるseedで3つのモデルを作り、予測を平均する
- [ ] 異なる `feature_fraction` で5つのモデルを作り、予測を平均する
- [ ] 訓練期間を少しずつずらした3つのモデルを作る（bagging over time）

### 金融ドメイン知識系
- [ ] モメンタム（ret_20d, ret_60d）とリバーサル（ret_5d）を分けて考える
- [ ] 低ボラティリティ銘柄にフォーカスする（高ボラ銘柄はノイズが多い）
- [ ] 出来高急増 × 株価上昇 の組み合わせ（機関投資家の買い集め）
- [ ] ボリンジャーバンド下限からの反発パターン

## 金融データ特有の罠

### サバイバーシップバイアス
上場廃止された銘柄がデータから消えていると、過去のパフォーマンスが良く見える。prepare.py が銘柄リストを固定しているため、2015年に上場していたが現在は廃止された銘柄が含まれない可能性がある。これは認識した上で「テスト期間の予測精度が真の実力」と割り切れ。

### レジームチェンジ
2015-2019年（アベノミクス後半）と 2020-2023年（コロナ〜金利上昇）では市場の性格が違う。訓練期間全体で学習すると、古いレジームに引っ張られる可能性がある。サンプル重み付け（直近を重視）が有効。

### 流動性の罠
出来高が極端に少ない銘柄がスコア上位に来ても、実際には売買できない。ただしこれは prepare.py/evaluate の管轄なので、train.py 側では気にしなくてよい。

### 決算発表バイアス
決算発表前後は株価が大きく動く。40営業日後リターンに決算の影響が混入する。これは避けようがないので、モデルが「決算前に買って決算後に持つ」パターンを学習してしまう可能性を認識しておけ。

## シンプルさの基準

**全ての変更は「改善の大きさ」対「複雑さの増加」で評価せよ。**

- top20_return が 0.001 改善したが、コードが20行増えた → **たぶん捨てるべき**
- top20_return が 0.001 改善して、コードが5行減った → **絶対に採用**
- top20_return が 0.01 改善して、コードが30行増えた → **おそらく採用**
- top20_return が変わらないが、コードが大幅にシンプルになった → **採用**

コードの複雑さはバグの温床。シンプルなモデルが複雑なモデルと同程度の性能なら、シンプルな方を選べ。

## クラッシュ時の対応ルール

1. **まず `tail -n 50 run.log` でエラーを確認**
2. **タイポ・import漏れ等の単純ミス** → 修正して再実行。コミットは上書き（amend）でよい。
3. **OOM（メモリ不足）** → モデルサイズを下げるか、バッチサイズを下げる。根本的に無理そうなら crash として記録し次へ。
4. **2回連続で同じ実験がクラッシュ** → その方向は諦める。crash を記録し、`git reset --hard HEAD~1` で戻る。
5. **意味不明なエラー** → エラーメッセージを読んで原因を推測。推測できなければ crash として記録し次へ。

---

## タイムアウト

- 通常の実行は1〜3分で完了するはず
- 5分を超えたらタイムアウトとして扱う
- 10分を超えた場合は `kill` して crash として記録
